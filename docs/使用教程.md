# STM32 Debug Studio 中文使用教程

本教程面向以下使用场景：

- 你要在 Windows 上运行 STM32 Debug Studio
- 你要在另一台电脑上用 VS Code + CMake 构建 STM32 工程
- 你要用 STM32 Debug Studio 下载 ELF 到开发板
- 你要在图形界面里进行断点调试、单步执行、查看调用栈、监视和修改变量

本文默认你使用的是基于 OpenOCD 的调试链路，包括 DAPLink 或兼容 CMSIS-DAP 的探针。

## 1. 程序定位

STM32 Debug Studio 是一个 Windows 桌面调试工具，目标是把以下几类操作集中到一个图形界面里：

- 管理 STM32 CMake 工程
- 调用 CMake 完成配置和构建
- 启动 OpenOCD
- 启动 arm-none-eabi-gdb
- 下载 ELF 到目标板
- 断点调试
- 查看调用栈
- 监视和修改全局变量

当前版本已经适合做基础下载与调试联调，但还不是完整的 Keil 等价替代品。寄存器窗口、内存窗口、SVD 外设树、RTOS 线程视图还在后续迭代范围内。

## 2. 运行前准备

在测试电脑上，至少需要准备以下外部工具：

- CMake
- OpenOCD
- Arm GNU Toolchain
- Ninja

其中：

- 如果你的 CMake 工程使用 Ninja 生成器，就必须有 Ninja
- 如果你已经有自己的 Makefile 或其他生成器，可以按项目实际情况调整

程序内部提供了“环境自检”按钮，用来检测这些工具是否已经能从 PATH 找到。

## 3. 如何获得程序

如果你在当前开发机上打包程序，推荐使用便携包方式：

1. 在项目根目录运行：

```powershell
npm.cmd run dist
```

2. 打包完成后，把下面这个文件复制到另一台 Windows 电脑：

- `release/STM32-Debug-Studio-win-unpacked.zip`

3. 在另一台电脑解压 ZIP。

4. 解压后的目录中会自带中文教程：

- `docs/使用教程.md`

5. 运行解压目录里的：

- `STM32 Debug Studio.exe`

如果你更喜欢安装器方式，也可以使用：

```powershell
npm.cmd run dist:installer
```

## 4. 如何启动程序

如果你在源码环境下直接运行：

```powershell
npm.cmd install
npm.cmd run build
npm.cmd start
```

如果你在开发模式下运行：

```powershell
npm.cmd run dev
```

## 5. 在另一台电脑上如何准备 STM32 工程

目标电脑上的 STM32 工程建议满足以下条件：

- 使用 VS Code 打开工程目录
- 工程由 CMake 管理
- 能在本机独立完成构建
- 构建后能产出 ELF 文件
- 使用 Debug 构建类型

推荐流程：

1. 在 VS Code 中打开 STM32 工程根目录。
2. 使用 CMake Tools 或命令行完成 Configure。
3. 执行 Debug 构建。
4. 确认生成了 ELF 文件，例如：

- `build/app.elf`
- `build/firmware.elf`
- `build/Debug/project.elf`

如果没有 ELF，STM32 Debug Studio 无法下载和调试。

## 6. 第一次打开程序后的操作顺序

建议严格按这个顺序操作：

1. 启动 STM32 Debug Studio
2. 点击“更多工具”里的“环境自检”
3. 点击“选择工程”
4. 在“工程设置”页填写工程参数
5. 点击“配置工程”
6. 点击“编译工程”
7. 先尝试“更多操作”里的“下载程序”
8. 再尝试“开始调试”

这样能把“构建问题”和“调试链路问题”分开排查。

## 7. 界面各区域怎么用

### 7.1 顶部状态区

这里主要显示：

- 当前平台
- 调试器连接状态
- 当前是“运行中”还是“已停住”

当前版本把这一块压缩成了更紧凑的状态条，主要用于快速确认调试链路是否连通，不再占用过多纵向空间。

如果调试成功连接，你会看到状态从“调试器空闲”变成“调试器已连接”。

### 7.2 顶部主操作区

主操作区只保留最常用的按钮：

- 选择工程
- 扫描工程

一些不需要频繁点按的操作被折叠到二级菜单里：

- “更多工具”里有“环境自检”和“生成 VS Code 配置”

当前版本还对这里做了两点界面优化：

- 顶部按钮、字体和间距整体缩小，避免上方两个模块占用过多空间
- “更多工具”菜单会浮在工作区上层显示，不会再被下方模块覆盖

这样做的目的，是避免在调试时为了点某个按钮还要滚轮找位置，同时减少顶部遮挡感，让代码区保留更多空间。

### 7.3 左侧面板

左侧改成了两个页签：

- 工程设置
- 源码列表

#### 工程设置

这是最关键的配置区。

为了避免调试时再用滚轮去找关键按钮，当前版本把最常用的操作前置到了基础参数区域后面：

- 配置工程
- 编译工程
- 开始调试

这样即使你不展开“高级调试参数”，也能先完成最常见的一轮下载和调试。

每个字段的含义如下。

#### 工程目录

STM32 工程根目录。

通常应指向含有 `CMakeLists.txt` 的目录。

#### 构建目录

构建目录。

常见值：

- `build`
- `out/build`
- `cmake-build-debug`

#### 生成器

CMake 生成器。

常见值：

- `Ninja`
- `Unix Makefiles`
- `MinGW Makefiles`

在 Windows 上推荐优先用 `Ninja`。

#### 构建类型

推荐填写：

- `Debug`

不要用高度优化的 Release 去做单步调试，否则断点和变量行为会很差。

#### 并行任务

并行编译线程数。

例如：

- `4`
- `8`
- `12`

#### 调试接口

这里用于选择当前使用的调试探针接口模板。

当前内置选项包括：

- 自定义 OpenOCD
- DAPLink
- CMSIS-DAP
- ST-Link
- J-Link

说明：

- 选择预设接口后，程序会自动补入对应的 OpenOCD interface cfg
- 选择“自定义 OpenOCD”时，说明你准备自己手写完整 OpenOCD 配置

#### 目标芯片/板卡

这里用于选择目标 MCU 或板卡模板。

当前内置选项包括：

- 自定义 target/board
- STM32F103 target
- STM32F407 target
- STM32H743 target
- STM32 Nucleo-F103RB board
- STM32 Nucleo-F446RE board

说明：

- 程序会把“调试接口”与“目标芯片/板卡”两个预设自动组合成一份 OpenOCD 启动配置
- 大多数常见场景下，不需要再手工填写 target cfg 或 board cfg
- 如果你选择“自定义 target/board”，就需要在下面的“额外 OpenOCD 配置”里补上目标相关配置

#### 调试配置

这里用于选择本次“开始调试”的启动策略。

当前内置选项包括：

- 下载并运行到 main
- 下载后停住
- 附加并停住
- 仅附加当前目标
- 自定义策略

说明：

- 选择预设后，下方“连接时下载 ELF”“连接后 reset halt”“自动运行到 main”会自动切换到对应组合
- 如果你手动改动这些开关，调试配置会自动变成“自定义策略”

#### 工具链文件

你的 STM32 CMake 工具链文件路径。

常见是 `.cmake` 文件。

如果项目本身已经在 `CMakeLists.txt` 或 preset 里固定好了，也可以留空或按项目约定填写。

#### CMake 额外参数

额外传给 CMake Configure 的参数。

例如：

- `-DMCU=STM32F103C8`
- `-DUSE_HAL_DRIVER=ON`

#### 构建目标

需要构建的目标名。

如果你的工程只有一个默认目标，可以留空。

#### OpenOCD 路径

OpenOCD 可执行文件名或完整路径。

常见值：

- `openocd`
- `C:\OpenOCD\bin\openocd.exe`

#### OpenOCD 配置

这里填写 OpenOCD 配置文件，可以使用：

- 分号分隔
- 或换行分隔

如果你选择的是“自定义 OpenOCD”，这里需要填写完整配置。

例如 DAPLink 或 CMSIS-DAP：

```text
interface/cmsis-dap.cfg; target/stm32f4x.cfg
```

例如 ST-Link：

```text
interface/stlink.cfg; target/stm32f4x.cfg
```

如果你的板卡有 board 配置，也可以用：

```text
board/st_nucleo_f4.cfg
```

如果你选择的是 DAPLink、CMSIS-DAP、ST-Link 或 J-Link 这类预设调试器，这里通常只需要再补目标板或 MCU 对应的 target/board 配置即可。

注意：`target/*.cfg` 必须与你的 STM32 型号匹配。

#### GDB 路径

GNU Arm GDB 的路径。

常见值：

- `arm-none-eabi-gdb`
- `C:\...\arm-none-eabi-gdb.exe`

#### ELF 文件

这是最重要的字段之一。

必须指向实际生成的 ELF 文件。

例如：

- `build/app.elf`
- `build/Debug/project.elf`

如果这里填错，下载和调试都会失败。

#### 连接时下载 ELF

勾选后，点击“开始调试”时会自动先把 ELF 下载到板子。

#### 连接后 reset halt

勾选后，连接成功后会执行 reset halt，让目标停住。

#### 自动运行到 main

勾选后，程序会尝试插入临时 main 断点并继续执行到 main。

#### 高级调试参数

OpenOCD 路径、额外 OpenOCD 配置、GDB 路径、调试启动开关等不常改的参数被折叠在“高级调试参数”里。

如果你只是重复调试同一个工程，通常不需要每次展开它。

#### 更多操作

“工程设置”页里只保留下面几个高频按钮：

- 配置工程
- 编译工程
- 开始调试

次要按钮被折叠进“更多操作”：

- 下载程序
- 停止调试

#### 源码列表

源码浏览也被独立成单独页签。

这样调试时不会再和工程参数、按钮堆在一起。你还可以在“源码列表”页先用筛选框定位文件，再点击打开。

### 7.4 中间代码区

中间区域保留为主要工作区：

- 源码
- 内存预览
- 寄存器说明
- 日志

左侧工程区、右侧监视区和中间代码区之间增加了可拖拽分隔条。

你可以直接拖动分隔条改变左右边栏宽度，在小屏或高分辨率显示器上快速切出更适合当前任务的布局。

调试控制按钮也做了分层：

- 第一层保留“继续”“暂停”“越过”
- 第二层放在“更多控制”里，包括“进入”“跳出”“复位”

当前版本还给代码区补上了两个更接近日常 IDE 的入口：

- 在代码里右键变量，可直接“添加到监视器”或“添加到示波器”
- 把光标停在函数名、变量名、宏名或类型名上，按 `F12` 或右键“转到定义”可快速跳转到定义位置

### 7.5 右侧信息区

右侧同样改成了两个页签：

- 监视示波
- 会话信息

#### 监视示波

这里集中放：

- 监视表页签
- 示波器页签

当前版本把右侧内容进一步收成 2 个二级页签：

右侧边栏和中间代码区之间同样支持拖拽改宽度；如果仍然觉得空间不够，可以把监视表或示波器弹成独立窗口。

##### 监视表

这里主要处理：

- 添加监视变量
- 查看树杈式树形监视表
- 展开结构体和 C++ class
- 双击值单元格直接修改变量
- 通过“更多监视”菜单刷新监视值或切到示波器
- 通过标题栏右上角按钮把监视表弹成独立窗口

##### 示波器

这里主要处理：

- 设置示波频率
- 把多个变量加入同一个示波窗口
- 调整时基窗口
- 切换纵轴自动缩放或手动量程
- 切换纵轴自动偏移或手动偏移量
- 开始/停止示波
- 查看实时曲线
- 查看纵轴数字刻度和每条曲线的图例
- 通过标题栏右上角按钮把示波器弹成独立窗口

这样做的目的，是把高频操作集中在一张表里，减少右侧堆叠卡片和滚动条，更接近 Keil 的监视体验。

#### 会话信息

这里集中放：

- 调用栈
- 当前会话概览
- 折叠后的环境诊断

## 8. Configure 和 Build 如何使用

### 配置工程

点击“配置工程”后，程序会调用 CMake，完成工程配置。

适合场景：

- 第一次打开新工程
- 改了 toolchain file
- 改了 CMake 参数
- 改了生成器

### 编译工程

点击“编译工程”后，程序会调用 CMake 构建。

建议在开始下载或调试之前先编译一次，确保 ELF 是最新版本。

## 9. 如何只下载程序到开发板

如果你现在只是想把程序烧录进去，不想马上开始调试，使用：

- “更多操作”里的“下载程序”

“下载程序”的行为是：

1. 启动 OpenOCD
2. 启动 GDB
3. 连接目标板
4. 执行下载
5. 尝试 reset run
6. 结束当前调试链路

这适合以下场景：

- 快速验证固件是否能正常启动
- 不需要停在断点
- 只想烧录最新版 ELF

## 10. 如何开始图形化调试

如果你要真正进入调试，使用：

- “开始调试”

“开始调试”的典型流程：

1. 启动 OpenOCD
2. 启动 GDB
3. 连接目标板
4. 按当前“调试配置”决定是否下载 ELF
5. 按当前“调试配置”决定是否 reset halt
6. 同步断点
7. 如果当前配置包含“运行到 main”，就继续执行到 main
8. 进入可调试状态

## 11. 如何打断点

在源码编辑器里，点击代码左侧的 gutter 区域即可切换断点。

当前版本做了更直观的左侧调试标记：

- 断点会显示为小红点
- 程序停住时，当前执行行左侧会显示双箭头
- 点击左侧的断点点位区、行号区或装饰区都可以切换断点

建议：

- 优先在可执行代码行打断点
- 不要在空行、声明行、宏定义区、纯注释行打断点
- 使用 Debug 构建，以减少断点飘移问题

## 12. 如何单步调试

连接成功后，可使用这些按钮：

- Continue：继续运行
- Pause：中断运行
- Step Over：单步越过
- Step Into：单步进入
- Step Out：单步跳出
- Reset：复位并重新停住

推荐调试节奏：

1. Start Debug
2. 停在 main 或某个初始化断点
3. Step Over 逐行确认初始化流程
4. 进入关键函数时用 Step Into
5. 跳出当前函数用 Step Out

## 13. 如何查看调用栈

右侧 Call Stack 面板会显示当前调用链。

你可以：

- 看当前停在哪个函数
- 看调用是从哪里进来的
- 点击某个栈帧跳转到对应源码

如果当前停下时没有正常显示调用栈，通常有几种原因：

- ELF 符号不完整
- 编译优化过高
- 当前停下的位置处于异常上下文

## 14. 如何监视和修改变量

### 添加监视变量

在“监视示波”页的输入框中输入表达式，例如：

- `globalCounter`
- `systemState`
- `adcBuffer[0]`

点击“添加”即可加入监视列表。

### 查看结构体和 C++ class

当前版本的监视区已经改成更接近专业 IDE 的树杈式树形表格，支持直接展开结构体、数组元素以及大多数 C++ class 成员。

使用方法：

1. 添加一个结构体、类对象或数组表达式
2. 在“监视表”里点击变量名前面的展开按钮
3. 展开后可以继续逐级查看子成员

说明：

- 左侧列显示变量名或成员名
- 右侧列显示当前值
- 树节点会像 Keil 一样显示分支线和折叠按钮，层级关系更直观
- 如果某个节点还能继续展开，说明 GDB 已识别到它有子成员

### 刷新变量

点击“更多监视”里的“刷新监视值”可重新读取当前监视变量值。

### 使用示波器查看变量曲线

当前版本的“监视示波”页已经整合了 Scope 面板，可以对多个已选中的监视变量做连续采样并同时查看曲线。

使用方法：

1. 先让程序停在断点、单步停住或 reset halt 之后的 Halted 状态
2. 切到“监视表”页签，在表格中点击一个变量
3. 切到“示波器”页签
4. 在顶部“加入变量”下拉框里选择一个变量，点击“加入曲线”
5. 如需对比多个变量，重复上一步，把多个变量加入同一个示波器
6. 在“示波频率”输入目标采样频率
7. 在“时基”里选择当前波形窗口长度，例如 `500 ms`、`1 s` 或 `5 s`
8. 如果希望纵轴跟随数据自动伸缩，保持“纵轴自动缩放”勾选
9. 如果希望自己固定显示范围，取消“纵轴自动缩放”，再填写“纵轴量程”
10. 如果希望纵轴中心跟随数据自动居中，保持“纵轴自动偏移”勾选
11. 如果希望自己指定纵轴偏移量，取消“纵轴自动偏移”，再填写“纵轴偏移”
12. 点击“开始示波”
13. 在右侧示波器区域查看多条曲线、图例、当前频率和纵轴数字刻度

如果右侧空间不够：

1. 点击示波器标题栏右上角的独立窗口按钮
2. 在弹出的独立示波器窗口中继续查看曲线

说明：

- 目标频率最高可以设置到 `1000 Hz`
- 界面会同时显示目标频率和实际达到的频率
- 可以同时把多个数值变量加入同一个示波器窗口做对比
- 每条曲线都有独立图例，纵轴会按当前采样值范围自动显示数字刻度
- 勾选“纵轴自动缩放”时，纵轴量程会随当前数据范围自动变化
- 取消“纵轴自动缩放”后，可以手工输入固定量程，方便做稳定对比
- 勾选“纵轴自动偏移”时，纵轴中心会自动围绕当前数据居中
- 取消“纵轴自动偏移”后，可以手工输入偏移量，让波形固定在你希望的纵轴位置
- 只有能解析成数值的标量变量才能画出曲线
- 如果变量值是结构体、字符串或复杂表达式，Watch 仍可显示文本值，但示波器不会出曲线
- 目标正在 Running 时，示波采样会自动暂停；目标重新 Halt 后会恢复

### 在代码区直接加入监视或示波

如果你已经在看源码，不想再切回右侧手工输入变量名，可以直接在代码区操作：

1. 把光标停在变量名上，或先选中一个表达式
2. 右键
3. 选择“添加到监视器”或“添加到示波器”

说明：

- “添加到监视器”会把当前符号直接加入右侧监视表
- “添加到示波器”会把当前符号加入示波器通道列表；如果监视表里还没有这个变量，也会一起补进监视表
- 对数组元素、成员表达式等，建议先精确选中表达式，再右键加入

### 在代码区跳转到定义

当前版本已经支持类似 VS Code 的代码跳转入口。

使用方法：

1. 把光标停在函数名、变量名、宏名或类型名上
2. 按 `F12`，或者右键后选择“转到定义”

说明：

- 跳转后，编辑器会自动打开对应文件并定位到定义所在行
- 当前版本使用工程源码扫描做定义查找，最适合 C/C++ 工程里常见的函数、变量、宏和类型定义

### 修改变量值

1. 在“监视表”中找到要修改的变量
2. 双击右侧“值”单元格
3. 输入新值后按 Enter，或直接点到别处提交

例如：

- 把 `globalCounter` 改成 `100`
- 把 `flag` 改成 `1`

注意：

- 变量必须在当前调试上下文中可访问
- 高优化构建下，某些局部变量可能不可见或行为异常
- 当前版本优先适合监视和修改全局变量、静态变量，以及能被 GDB 正常解析的结构体/class 成员

## 16. DAPLink / CMSIS-DAP 配置建议

如果你使用的是 DAPLink 或其他 CMSIS-DAP 探针，可以从下面这种写法开始：

```text
interface/cmsis-dap.cfg; target/stm32f1x.cfg
```

或者：

```text
interface/cmsis-dap.cfg; target/stm32f4x.cfg
```

你需要根据自己的 MCU 型号替换 target 文件。

常见思路：

- STM32F1 系列：`target/stm32f1x.cfg`
- STM32F4 系列：`target/stm32f4x.cfg`
- STM32H7 系列：`target/stm32h7x.cfg`

如果不确定型号配置，先去你本机的 OpenOCD scripts 目录里查 target 文件名。

## 18. 常见问题排查

### 问题 1：环境自检提示某个工具缺失

处理方式：

- 确认工具已经安装
- 确认命令已经加入 PATH
- 如果没加 PATH，就填可执行文件完整路径

### 问题 2：下载程序失败

常见原因：

- OpenOCD config 填错
- 目标芯片型号和 target cfg 不匹配
- ELF 路径错误
- 开发板未连接或探针未识别
- 端口被占用

### 问题 3：开始调试后进不了断点

常见原因：

- 编译不是 Debug
- 代码被优化
- 断点打在不可执行行
- 用错了 ELF
- 下载的程序和当前源码不一致

### 问题 4：变量读不到或改不了

常见原因：

- 变量作用域不可见
- 局部变量已被优化掉
- 程序当前并没有停在适合读取变量的位置

### 问题 5：示波器没有曲线或频率达不到预期

优先检查：

- 当前是否已经停在 Halted 状态
- 选中的变量是否是单个数值变量，而不是结构体或字符串
- GDB 读取该变量本身是否稳定
- OpenOCD、探针和目标板连接是否稳定
- 目标频率虽然可设置到 `1000 Hz`，但实际频率取决于 GDB MI 往返速度、探针链路和变量求值开销

### 问题 6：OpenOCD 启动失败

优先检查：

- OpenOCD Path 是否正确
- OpenOCD Config 是否正确
- 探针驱动是否正常
- 板子是否上电

## 19. 推荐使用习惯

为了让调试更稳定，建议按下面的习惯使用：

- 统一使用 Debug 构建
- 每次调试前先编译工程
- 先用“下载程序”验证烧录链路
- 再用“开始调试”验证断点和变量链路
- 常用按钮直接在主界面点，低频操作放到“更多工具”“更多操作”“更多控制”里
- 需要观察波形时，优先选用简单的全局数值变量挂到示波器
- 使用环境自检排除目标机缺工具的问题
- 保持 ELF 和当前源码版本一致
