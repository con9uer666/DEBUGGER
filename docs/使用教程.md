# STM32 Debug Studio 中文使用教程

本教程面向以下使用场景：

- 你要在 Windows 上运行 STM32 Debug Studio
- 你要在另一台电脑上用 VS Code + CMake 构建 STM32 工程
- 你要用 STM32 Debug Studio 下载 ELF 到开发板
- 你要在图形界面里进行断点调试、单步执行、查看调用栈、监视和修改变量

本文默认你使用的是基于 OpenOCD 的调试链路，包括 DAPLink 或兼容 CMSIS-DAP 的探针。

## 1. 程序定位

STM32 Debug Studio 是一个 Windows 桌面调试工具，目标是把以下几类操作集中到一个图形界面里：

- 管理 STM32 CMake 工程
- 调用 CMake 完成配置和构建
- 启动 OpenOCD
- 启动 arm-none-eabi-gdb
- 下载 ELF 到目标板
- 断点调试
- 查看调用栈
- 监视和修改全局变量

当前版本已经适合做基础下载与调试联调，但还不是完整的 Keil 等价替代品。寄存器窗口、内存窗口、SVD 外设树、RTOS 线程视图还在后续迭代范围内。

## 2. 运行前准备

在测试电脑上，至少需要准备以下外部工具：

- CMake
- OpenOCD
- Arm GNU Toolchain
- Ninja

其中：

- 如果你的 CMake 工程使用 Ninja 生成器，就必须有 Ninja
- 如果你已经有自己的 Makefile 或其他生成器，可以按项目实际情况调整

程序内部提供了“环境自检”按钮，用来检测这些工具是否已经能从 PATH 找到。

## 3. 如何获得程序

如果你在当前开发机上打包程序，推荐使用便携包方式：

1. 在项目根目录运行：

```powershell
npm.cmd run dist
```

2. 打包完成后，把下面这个文件复制到另一台 Windows 电脑：

- `release/STM32-Debug-Studio-win-unpacked.zip`

3. 在另一台电脑解压 ZIP。

4. 解压后的目录中会自带中文教程：

- `docs/使用教程.md`

5. 运行解压目录里的：

- `STM32 Debug Studio.exe`

如果你更喜欢安装器方式，也可以使用：

```powershell
npm.cmd run dist:installer
```

## 4. 如何启动程序

如果你在源码环境下直接运行：

```powershell
npm.cmd install
npm.cmd run build
npm.cmd start
```

如果你在开发模式下运行：

```powershell
npm.cmd run dev
```

## 5. 在另一台电脑上如何准备 STM32 工程

目标电脑上的 STM32 工程建议满足以下条件：

- 使用 VS Code 打开工程目录
- 工程由 CMake 管理
- 能在本机独立完成构建
- 构建后能产出 ELF 文件
- 使用 Debug 构建类型

推荐流程：

1. 在 VS Code 中打开 STM32 工程根目录。
2. 使用 CMake Tools 或命令行完成 Configure。
3. 执行 Debug 构建。
4. 确认生成了 ELF 文件，例如：

- `build/app.elf`
- `build/firmware.elf`
- `build/Debug/project.elf`

如果没有 ELF，STM32 Debug Studio 无法下载和调试。

## 6. 第一次打开程序后的操作顺序

建议严格按这个顺序操作：

1. 启动 STM32 Debug Studio
2. 点击“环境自检”
3. 点击“选择工程目录”
4. 填写 Project Profile
5. 点击 Configure
6. 点击 Build
7. 先尝试 Program Device
8. 再尝试 Start Debug

这样能把“构建问题”和“调试链路问题”分开排查。

## 7. 界面各区域怎么用

### 7.1 顶部状态区

这里主要显示：

- 当前平台
- 调试器连接状态
- 当前是 Running 还是 Halted

如果调试成功连接，你会看到调试状态从 Idle 变成 Connected。

### 7.2 Project Profile 区域

这是最关键的配置区。

每个字段的含义如下。

#### Project Root

STM32 工程根目录。

通常应指向含有 `CMakeLists.txt` 的目录。

#### Build Dir

构建目录。

常见值：

- `build`
- `out/build`
- `cmake-build-debug`

#### Generator

CMake 生成器。

常见值：

- `Ninja`
- `Unix Makefiles`
- `MinGW Makefiles`

在 Windows 上推荐优先用 `Ninja`。

#### Build Type

推荐填写：

- `Debug`

不要用高度优化的 Release 去做单步调试，否则断点和变量行为会很差。

#### Jobs

并行编译线程数。

例如：

- `4`
- `8`
- `12`

#### Toolchain File

你的 STM32 CMake 工具链文件路径。

常见是 `.cmake` 文件。

如果项目本身已经在 `CMakeLists.txt` 或 preset 里固定好了，也可以留空或按项目约定填写。

#### CMake Extra Args

额外传给 CMake Configure 的参数。

例如：

- `-DMCU=STM32F103C8`
- `-DUSE_HAL_DRIVER=ON`

#### Build Target

需要构建的目标名。

如果你的工程只有一个默认目标，可以留空。

#### OpenOCD Path

OpenOCD 可执行文件名或完整路径。

常见值：

- `openocd`
- `C:\OpenOCD\bin\openocd.exe`

#### OpenOCD Config

这里填写 OpenOCD 配置文件，可以使用：

- 分号分隔
- 或换行分隔

例如 DAPLink 或 CMSIS-DAP：

```text
interface/cmsis-dap.cfg; target/stm32f4x.cfg
```

例如 ST-Link：

```text
interface/stlink.cfg; target/stm32f4x.cfg
```

如果你的板卡有 board 配置，也可以用：

```text
board/st_nucleo_f4.cfg
```

注意：`target/*.cfg` 必须与你的 STM32 型号匹配。

#### GDB Path

GNU Arm GDB 的路径。

常见值：

- `arm-none-eabi-gdb`
- `C:\...\arm-none-eabi-gdb.exe`

#### ELF File

这是最重要的字段之一。

必须指向实际生成的 ELF 文件。

例如：

- `build/app.elf`
- `build/Debug/project.elf`

如果这里填错，下载和调试都会失败。

#### Connect 时下载 ELF

勾选后，点击 Start Debug 时会自动先把 ELF 下载到板子。

#### Connect 后 reset halt

勾选后，连接成功后会执行 reset halt，让目标停住。

#### 自动运行到 main

勾选后，程序会尝试插入临时 main 断点并继续执行到 main。

## 8. Configure 和 Build 如何使用

### Configure

点击 Configure 后，程序会调用 CMake，完成工程配置。

适合场景：

- 第一次打开新工程
- 改了 toolchain file
- 改了 CMake 参数
- 改了生成器

### Build

点击 Build 后，程序会调用 CMake 构建。

建议在开始下载或调试之前先 Build 一次，确保 ELF 是最新版本。

## 9. 如何只下载程序到开发板

如果你现在只是想把程序烧录进去，不想马上开始调试，使用：

- `Program Device`

Program Device 的行为是：

1. 启动 OpenOCD
2. 启动 GDB
3. 连接目标板
4. 执行下载
5. 尝试 reset run
6. 结束当前调试链路

这适合以下场景：

- 快速验证固件是否能正常启动
- 不需要停在断点
- 只想烧录最新版 ELF

## 10. 如何开始图形化调试

如果你要真正进入调试，使用：

- `Start Debug`

Start Debug 的典型流程：

1. 启动 OpenOCD
2. 启动 GDB
3. 连接目标板
4. 如果勾选了自动下载，就先下载 ELF
5. 同步断点
6. 如果勾选了“自动运行到 main”，就继续执行到 main
7. 进入可调试状态

## 11. 如何打断点

在源码编辑器里，点击行号左侧的 gutter 区域即可切换断点。

建议：

- 优先在可执行代码行打断点
- 不要在空行、声明行、宏定义区、纯注释行打断点
- 使用 Debug 构建，以减少断点飘移问题

## 12. 如何单步调试

连接成功后，可使用这些按钮：

- Continue：继续运行
- Pause：中断运行
- Step Over：单步越过
- Step Into：单步进入
- Step Out：单步跳出
- Reset：复位并重新停住

推荐调试节奏：

1. Start Debug
2. 停在 main 或某个初始化断点
3. Step Over 逐行确认初始化流程
4. 进入关键函数时用 Step Into
5. 跳出当前函数用 Step Out

## 13. 如何查看调用栈

右侧 Call Stack 面板会显示当前调用链。

你可以：

- 看当前停在哪个函数
- 看调用是从哪里进来的
- 点击某个栈帧跳转到对应源码

如果当前停下时没有正常显示调用栈，通常有几种原因：

- ELF 符号不完整
- 编译优化过高
- 当前停下的位置处于异常上下文

## 14. 如何监视和修改变量

### 添加监视变量

在 Watch 区输入表达式，例如：

- `globalCounter`
- `systemState`
- `adcBuffer[0]`

点击 Add 即可加入监视列表。

### 刷新变量

点击 Refresh 可重新读取当前监视变量值。

### 使用示波器查看变量曲线

当前版本的 Watch 区已经整合了 Scope 面板，可以对一个已选中的监视变量做连续采样并查看曲线。

使用方法：

1. 先让程序停在断点、单步停住或 reset halt 之后的 Halted 状态
2. 在 Watch 列表中点击一个变量
3. 在 `Scope Hz` 输入目标采样频率
4. 点击 `Start Scope`
5. 在右侧示波器区域查看曲线、最新值、最近样本时间和当前频率

说明：

- 当前一次只支持把一个变量挂到示波器
- 目标频率最高可以设置到 `1000 Hz`
- 界面会同时显示目标频率和实际达到的频率
- 只有能解析成数值的标量变量才能画出曲线
- 如果变量值是结构体、字符串或复杂表达式，Watch 仍可显示文本值，但示波器不会出曲线
- 目标正在 Running 时，示波采样会自动暂停；目标重新 Halt 后会恢复

### 修改变量值

1. 在 Watch 列表中点击要修改的变量
2. 在 New Value 输入新值
3. 点击 Write Variable

例如：

- 把 `globalCounter` 改成 `100`
- 把 `flag` 改成 `1`

注意：

- 变量必须在当前调试上下文中可访问
- 高优化构建下，某些局部变量可能不可见或行为异常
- 当前版本更适合监视和修改全局变量、静态变量、简单表达式

## 15. 如何生成 VS Code 配置

点击：

- `生成 VS Code 配置`

程序会为当前 STM32 工程生成：

- `.vscode/tasks.json`
- `.vscode/launch.json`
- `.vscode/settings.json`

这适合以下情况：

- 你希望同一个工程既能在 STM32 Debug Studio 中调试，也能在 VS Code 中直接用 Cortex-Debug 调试
- 你想把当前调试参数固化到工程目录里

## 16. DAPLink / CMSIS-DAP 配置建议

如果你使用的是 DAPLink 或其他 CMSIS-DAP 探针，可以从下面这种写法开始：

```text
interface/cmsis-dap.cfg; target/stm32f1x.cfg
```

或者：

```text
interface/cmsis-dap.cfg; target/stm32f4x.cfg
```

你需要根据自己的 MCU 型号替换 target 文件。

常见思路：

- STM32F1 系列：`target/stm32f1x.cfg`
- STM32F4 系列：`target/stm32f4x.cfg`
- STM32H7 系列：`target/stm32h7x.cfg`

如果不确定型号配置，先去你本机的 OpenOCD scripts 目录里查 target 文件名。

## 17. 另一台电脑上的完整测试流程

下面是一套推荐的真实测试流程。

### 第一步：准备测试机环境

在测试机安装：

- VS Code
- CMake
- Arm GNU Toolchain
- OpenOCD
- Ninja

### 第二步：在测试机构建 STM32 工程

1. 用 VS Code 打开 STM32 项目
2. 执行 Configure
3. 执行 Build
4. 找到 ELF 文件

### 第三步：运行 STM32 Debug Studio

1. 解压便携包
2. 运行 `STM32 Debug Studio.exe`
3. 点击“环境自检”

### 第四步：配置调试参数

至少填这些：

- Project Root
- OpenOCD Config
- GDB Path
- ELF File

### 第五步：先下载，再调试

推荐顺序：

1. Program Device
2. 确认板子程序已正常运行
3. Start Debug
4. 在 `main` 或业务函数打断点
5. 执行单步和变量监视

## 18. 常见问题排查

### 问题 1：环境自检提示某个工具缺失

处理方式：

- 确认工具已经安装
- 确认命令已经加入 PATH
- 如果没加 PATH，就填可执行文件完整路径

### 问题 2：Program Device 失败

常见原因：

- OpenOCD config 填错
- 目标芯片型号和 target cfg 不匹配
- ELF 路径错误
- 开发板未连接或探针未识别
- 端口被占用

### 问题 3：Start Debug 后进不了断点

常见原因：

- 编译不是 Debug
- 代码被优化
- 断点打在不可执行行
- 用错了 ELF
- 下载的程序和当前源码不一致

### 问题 4：变量读不到或改不了

常见原因：

- 变量作用域不可见
- 局部变量已被优化掉
- 程序当前并没有停在适合读取变量的位置

### 问题 5：示波器没有曲线或频率达不到预期

优先检查：

- 当前是否已经停在 Halted 状态
- 选中的变量是否是单个数值变量，而不是结构体或字符串
- GDB 读取该变量本身是否稳定
- OpenOCD、探针和目标板连接是否稳定
- 目标频率虽然可设置到 `1000 Hz`，但实际频率取决于 GDB MI 往返速度、探针链路和变量求值开销

### 问题 6：OpenOCD 启动失败

优先检查：

- OpenOCD Path 是否正确
- OpenOCD Config 是否正确
- 探针驱动是否正常
- 板子是否上电

## 19. 推荐使用习惯

为了让调试更稳定，建议按下面的习惯使用：

- 统一使用 Debug 构建
- 每次调试前先 Build
- 先 Program Device 验证烧录链路
- 再 Start Debug 验证断点和变量链路
- 需要观察波形时，优先选用简单的全局数值变量挂到示波器
- 使用环境自检排除目标机缺工具的问题
- 保持 ELF 和当前源码版本一致

## 20. 当前版本最适合的用途

当前版本最适合以下任务：

- STM32 固件烧录
- OpenOCD + GDB 的基础图形化调试
- 断点调试
- 调用栈查看
- 全局变量监视与修改
- 单变量高频示波曲线观察
- 为 VS Code 工程生成调试配置

如果你后续要把它继续推进到更接近 Keil 的体验，下一步最值得补的是：

- 内存窗口
- 寄存器窗口
- SVD 外设窗口
- RTOS 线程窗口
- 更完整的下载配置和烧录选项
